## 问题陈述
+
当前项目为单文件、纯 HTML + 内联 CSS/JS 的画布应用。此结构在如下方面受限：
- 缺少组件化与模块边界，难以维护和扩展
- 样式复用与主题能力弱，难以统一 UI 规范
- 无设计系统与交互范式，影响一致性与可用性
- 无现代构建链路（Tree-shaking、按需加载、类型检查），性能与质量管控不足
+
需要将项目重构为标准前端工程，采用 React 19 + TailwindCSS + Arco Design，以获得组件化、现代构建与成熟设计系统支持，同时保留像素画编辑的核心能力。
+
## 行为契约
+
### 前置条件
- Node.js 环境可用；允许引入构建工具（Vite）与 npm 依赖
- 仍为纯前端应用，无后端依赖
- 迁移后保留现有核心功能：像素级渲染、工具栏操作、图层/预览、导入导出
+
### 后置条件
- P1：构建产物可作为静态资源部署，开发时可使用本地 Dev Server
- P2：UI 以组件化形式运行，交互与画布功能与现状等价或更佳
- P3：故障时提供明确错误提示，构建失败不产生脏产物
+
### 不变量
- 保持像素画渲染的清晰度（CSS `image-rendering: pixelated`）
- 保持暗色主题与整体观感
- 不引入不必要的重量级依赖与服务端耦合
+

## 提案
+
技术选型与总体方案：
- 构建工具：Vite（快速、轻量、良好 HMR，支持 React 19）
- 语言：TypeScript（类型安全与可维护性），ES 模块
- UI 框架：React 19（采用函数组件与 Hooks，逐步迁移状态与事件）
- 设计系统：Arco Design（桌面端优秀交互与丰富组件，按需引入）
- 样式：TailwindCSS（原子类与设计令牌，提高样式一致性与开发效率）
- 目录结构：`src/` 下分层（components/pages/hooks/lib/styles），`public/` 静态资源
- 样式与主题：保留暗色主题，融合 Arco 主题与 Tailwind 基线（通过自定义 CSS 变量/配置）
- 画布渲染：封装 `CanvasWorkspace` 组件，确保 `image-rendering: pixelated` 与像素对齐
- 状态管理：React Hooks（必要时使用 `useReducer`），避免引入额外状态库
- 可访问性：遵循 Arco 与标准 WAI-ARIA 建议，键盘交互保持与增强
+
迁移映射：
- 工具栏 → `Toolbar`（Arco Button/Dropdown + Tailwind 布局）
- 图层/预览 → `LayersPanel`、`PreviewPanel`
- 画布工作区 → `CanvasWorkspace`（封装绘制/事件处理）
- 导入/导出 → `ImportExport`（文件选择与下载）
- 全局布局 → `AppLayout`（Header/Side/Content）
+
性能与质量：
- 按需引入 Arco 组件，利用 Vite 与 ESBuild Tree-shaking
- Tailwind 的 `content` 精确匹配，避免未使用样式
- 类型检查通过 TS，最小化运行时错误
+
## 备选方案与取舍
- CRA：已过时且生态下行，构建慢，放弃
- Next.js：适合 SSR/路由需求，当前纯前端单页 + 画布场景过重，暂不采用
- 无框架继续演进：维护成本高、扩展受限，无法获得设计系统与工程化优势
+
